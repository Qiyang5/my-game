# 工位求生改版设计文档

- 日期：2026-03-31
- 状态：已批准
- 方向：保留题材，重做核心交互与页面表达

## 1. 这次改版要解决什么

当前版本的问题不是“内容少”，而是“核心循环没有成立”。

- 玩家看到的是 `点按 / 长按 / 拖拽`，而不是“我现在要做什么”
- 页面没有形成一个一眼可懂、同时又持续失控的“工位灾难现场”
- 单局中缺少连锁压力和强烈反馈，因此不够慌、不够爽、不够搞笑

这次改版的目标不是继续往旧结构上加事件，而是把《工位求生》重做成一个“夸张混乱、语义清晰、第一眼就像游戏”的移动端爆屏求生游戏。

## 2. 新版核心体验

玩家扮演一个正在工位上极限求生的打工人。整张屏幕就是工位桌面：电脑、手机、群聊、老板靠近提示、会议弹窗、报错窗口会不断争抢注意力。玩家面对的不是抽象手势，而是一个个具体行为决策：

- 接电话
- 按住装忙
- 把摸鱼页面拖进回收站
- 立刻回一句“收到”

玩家知道自己“要做什么”，但忙不过来；乐趣来自高压、多线、不断失控的救火过程。

## 3. 页面视觉方向

### 3.1 页面气质

采用 `夸张混乱` 的主基调：

- 不是简洁卡片 UI，而是爆屏工位
- 弹窗可以大、歪、抖、覆盖彼此
- 老板靠近要有明显压迫感
- 手机、电脑、群聊、会议提醒要同时存在争夺注意力

第一眼应该让玩家感觉：“这工位马上要炸了。”

### 3.2 主布局

- 顶部：倒计时、生存值、分数
- 中央主区：电脑屏幕，是主要危机发生地
- 左下或右下：手机区域，负责电话/即时消息骚扰
- 边缘层：老板靠近、群聊消息、会议提醒、报错等插入式危机
- 覆盖层：高压时出现红色警示、抖动、危险边框

## 4. 交互设计原则

### 4.1 底层手势保留，表层语义重做

底层仍然可以使用点击、长按、拖拽实现，但玩家不能感知到“手势类型”，只能感知到“行为含义”。

错误示例：

- 点击处理
- 长按处理
- 拖拽处理

正确示例：

- 接电话
- 按住装忙
- 拖走摸鱼窗口
- 立刻进会

### 4.2 事件只保留一个最直觉的解法

每个事件应该让玩家第一眼知道自己要干嘛。失败来自来不及、优先级判断失误、连锁灾难，而不是机制理解门槛。

## 5. 核心事件重构

新版首版保留 4~5 个高辨识度事件，每个都在页面上“长出来”。

### 5.1 老板靠近

- 视觉：老板头像、脚步、红色靠近提示从边缘压进来
- 行为：`按住装忙`
- 目的：制造持续压迫，打断别的操作

### 5.2 群聊轰炸

- 视觉：电脑右下角连续冒出消息气泡，关键消息带红点或 @
- 行为：`点关键消息`
- 目的：制造多目标筛选压力

### 5.3 外卖电话

- 视觉：手机震动、来电弹窗闪烁
- 行为：`接电话` 或 `挂断`
- 目的：让玩家在工位内外注意力来回拉扯

### 5.4 摸鱼暴露

- 视觉：电脑主屏显示短视频、购物页、聊天八卦页
- 行为：`拖进回收站` / `拖到后台`
- 目的：提供最直观、最有喜感的危险操作

### 5.5 临时会议 / 电脑报错

- 视觉：倒计时入会弹窗或遮挡型报错窗口
- 行为：`立刻进会` / `狂点修复`
- 目的：形成必须优先处理的硬压力

## 6. 单局节奏

单局仍然固定为 45 秒，但节奏从“单个事件处理”改为“逐步失控的多线救火”。

- 0~5 秒：玩家熟悉工位布局和第一个事件
- 5~15 秒：双线冲突开始
- 15~30 秒：电脑 + 手机 + 边缘告警同时工作
- 30~45 秒：进入爆屏阶段，必须取舍，明显感觉要崩

## 7. 失败与反馈

失败不能只表现为数字变化，而要表现为“现场失控”。

- 老板贴脸
- 手机狂震变红
- 群消息堆叠遮挡
- 报错窗口抖动放大
- 危险边框与音效同步升级

玩家应该能明确感受到：不是系统偷偷扣了分，而是“我真的翻车了”。

## 8. 上头机制

新版不是靠复杂成长，而是靠：

- 语义清晰，所以每次失败都知道自己错在哪
- 压力真实，所以每次翻车都觉得“差一点就能救回来”
- 结算有梗，所以失败也值得看完

目标是让玩家失败后立刻产生“这把我处理顺序再好一点就行”的冲动。

## 9. 结算页目标

结算页要从“分数页”升级为“职场翻车报告”。

- 展示得分、存活时长、处理成功数
- 给玩家一个明确称号
- 给出一句可截图传播的吐槽文案

例如：

- “你成功顶住了 9 次工位危机，但还是倒在老板回头杀。”
- “称号：已读乱回王”

## 10. 首版改版范围

优先做：

- 重做主界面布局
- 所有交互改为语义化
- 重做事件表现方式
- 加强失败反馈
- 改造结算页文案气质

暂不优先：

- 在线排行榜
- 分享图自动生成
- 大量新事件包
- 复杂角色成长
- 皮肤与装饰系统

## 11. 技术影响

当前技术栈仍可继续使用 Flutter，不需要更换。

这次改版主要影响：

- `AppShell` 与主界面层级
- `GameScreen` 的布局与交互映射
- 事件模型展示方式
- 结算页文案与表现
- 相关 widget / flow 测试

状态控制和基础事件调度可以复用，但 UI 表达、操作语义和局内反馈需要明显重构。
